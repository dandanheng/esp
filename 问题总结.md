现象

  切换到 HuoshanDoubleStreamTTS 后，音箱被唤醒但不响应语音。日志显示"大模型收到用户消息"之后完全卡住 20+
  秒，直到连接超时断开。

  根本原因

  两个线程同时争抢事件循环，但其中一个没有超时保护：

  chat() 线程（executor）                    tts_text_priority_thread（daemon）
      │                                              │
      ├─ 放 FIRST 消息到 TTS 队列                    ├─ 取出 FIRST 消息
      │                                              │
      ├─ asyncio.run_coroutine_threadsafe(           ├─ asyncio.run_coroutine_threadsafe(
      │    query_memory, loop                        │    start_session, loop      ← 连 Huoshan WebSocket
      │  ).result()     ← ❌ 无 timeout 死等          │  ).result(timeout=15)

  start_session 要连接远端 WebSocket（wss://openspeech.bytedance.com），这个网络操作本身是 async
  的、不阻塞事件循环，query_memory（nomem，直接返回 ""）理应被事件循环调度到立刻完成。

  但实际上卡住了。目前还有一个未解释的点：按 asyncio 机制，query_memory 应能在 start_session 等待网络 I/O
  的间隙被调度到，但实验表明并不成功。

  排除的可能性

  - Kimi API 本身 ✅ 正常（直接测试响应速度 < 1s）
  - Huoshan WebSocket 连接 ✅ 正常（单独测试能连上）
  - TTS 初始化 ✅ 正常（单独测试成功）
  - enable_ws_reuse: false ❌ 没有解决问题

  可能的真正原因（待验证）

  1. start_session 里有一段同步执行代码（未 await）在事件循环上占用 CPU，导致 query_memory 被延迟调度，而
  future.result() 无超时就死等
  2. Huoshan 的 WebSocket 握手在特定网络环境下无响应（但 TCP 已连通），导致 websockets.connect() 长时间挂住
  3. _start_monitor_tts_response 任务一旦创建，在某些情况下同步执行了大量代码
