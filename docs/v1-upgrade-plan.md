# 智能音箱 V1 升级计划

## 1. 文档目标

基于当前音箱的真实状态，给出一个可以直接执行的 V1 升级计划。

这份计划只解决两个问题：

1. 孩子已经反馈“它比较笨”，要先把交互挫败感降下来。
2. 当前没有稳定的睡前使用习惯，V1 不应该押在睡前陪伴，而应该切到白天高频小游戏和轻陪伴。

---

## 2. 当前实际情况

### 2.1 当前部署基线

根据项目现状，当前主链路基线应按下面这套来做：

- 设备：ESP32-S3-BOX-3
- LLM：当前实际使用 Kimi 接入方案
- ASR：FunASR 本地
- VAD：SileroVAD 本地
- TTS：EdgeTTS
- HuoshanDoubleStreamTTS：仍处于待修复状态，**不作为 V1 基线**

参考资料：

- `PROJECT_PIPELINE.md`
- `问题总结.md`
- `xiaozhi-esp32-server/main/xiaozhi-server/config.yaml`

### 2.2 当前关键限制

1. **不能继续追求“大而全”**
   现在的问题不是功能太少，而是“已有能力在儿童场景下不好用”。

2. **当前默认 prompt 不适合 6 岁男孩场景**
   `config.yaml` 里的默认角色是“台湾腔 00 后女生”，更像成人闲聊人格，不像儿童玩伴。

3. **当前默认 Memory 是 `nomem`**
   这会直接导致“记不住孩子刚说过什么”，很容易被孩子评价为“笨”。

4. **当前 TTS 稳定性优先级高于音色丰富度**
   EdgeTTS 已可用，Huoshan 先不要进入 V1 主路径。

## 3. V1 产品定义

### 3.1 V1 不做什么

- 不做开放式长聊天
- 不做睡前陪伴主场景
- 不做复杂多 Agent 编排
- 不做完整教育体系
- 不做依赖 Huoshan 的音色方案

### 3.2 V1 做什么

把音箱做成一个 **“6 岁男孩的语音互动玩伴”**。

V1 聚焦两个方向：

1. **短时小游戏**
   猜谜语、口算闯关、兴趣主题问答。

2. **轻陪伴互动**
   孩子说得不清楚也能接住，能顺着玩，能记得最近在玩什么。

### 3.3 V1 成功标准

1. 孩子愿意主动喊它玩。
2. 单次互动能稳定持续 2 到 5 分钟。
3. “它比较笨”的反馈明显下降。
4. 家长能感知到它反应更快、更懂孩子、没那么死板。

---

## 4. 核心判断

孩子口中的“笨”，大概率不是在评价模型知识量，而是在评价下面几种体验：

1. 听不懂
2. 反应慢
3. 不会接话
4. 记不住
5. 回答像老师，不像玩伴

所以 V1 的优化优先级不是“换更强模型”，而是：

1. **更快开口**
2. **更像儿童玩伴**
3. **更会接球**
4. **更有短期记忆**
5. **更适合高频小游戏**

## 5. 两周落地计划

## Phase 0：稳定基线确认（半天）

### 目标

先把技术基线定死，避免一边修 Huoshan 一边做 V1，导致验证失真。

### 动作

1. 固定 V1 基线为：
   - TTS：`EdgeTTS`
   - ASR：`FunASR`
   - VAD：`SileroVAD`
   - Memory：从 `nomem` 切到最小可用短记忆方案

2. 所有 V1 验证先不依赖 Huoshan。

### 需要关注的文件

- `xiaozhi-esp32-server/main/xiaozhi-server/config.yaml`
- `xiaozhi-esp32-server/main/xiaozhi-server/data/.config.yaml`

### 交付物

- 一份专门给 V1 用的 `data/.config.yaml` 覆盖配置

---

## Phase 1：先解决“比较笨”的感知（1 到 2 天）

### 目标

不新增复杂功能，先把交互体验从“勉强能用”拉到“像个玩伴”。

### 任务 1：替换儿童场景 prompt

#### 要做什么

把当前默认成人闲聊人设替换成儿童场景 prompt，核心要求：

- 角色：像 6 岁男孩的玩伴，不像老师
- 语言：短句、口语、少解释
- 风格：多接球，多给选项
- 约束：不要长篇科普，不要说教

#### 建议规则

- 每次回复尽量控制在 1 到 3 句
- 单句尽量不超过 20 字
- 优先使用问题引导
- 听不懂时给两个选项，不重复追问“你再说一遍”

#### 落点文件

- 首选：`xiaozhi-esp32-server/main/xiaozhi-server/data/.config.yaml`
- 如需模板化增强：`core/utils/prompt_manager.py`
- 如需统一模板：`agent-base-prompt.txt`

### 任务 2：压缩输出长度，优先提升首包响应

#### 要做什么

- 把聊天输出控制得更短
- 能先说一句就先说，不等长段内容凑齐

#### 建议参数

- `max_tokens` 先降到 `120~200`
- 温度保持中低，优先稳定

#### 落点文件

- `xiaozhi-esp32-server/main/xiaozhi-server/config.yaml`
- `core/providers/llm/openai/openai.py`

### 任务 3：加“模糊接球”逻辑

#### 要做什么

当孩子说得不清楚时，不让系统频繁失败。

先把用户输入粗分成这几类：

- 想玩游戏
- 想聊天
- 想听题目
- 没听清 / 不确定

然后统一用引导式回复继续，不追求精确理解原句。

#### 实现建议

第一版不要做复杂分类模型，直接在 prompt 和会话状态里做轻路由。

#### 落点文件

- `core/connection.py`
- `core/handle/textHandle.py`
- `core/handle/textMessageProcessor.py`
- 如需意图扩展：`core/providers/intent/function_call/function_call.py`

### Phase 1 验收标准

- 孩子连续说 5 轮，不明显卡死
- 不频繁出现答非所问
- 回复长度明显缩短
- 家长主观感觉“没那么笨了”

---

## Phase 2：上线 3 个高频小游戏（3 到 5 天）

### 目标

让孩子“随时有得玩”，而不是只能闲聊。

### V1 只上这 3 个游戏

1. **猜谜语**
   - 最容易启动
   - 反馈快
   - 适合 5 到 8 轮短回合

2. **口算闯关**
   - 10 以内加减法起步
   - 每次 5 题
   - 易于观察进步

3. **兴趣主题问答**
   - 恐龙 / 动物 / 交通工具 三选一
   - 一问一答式
   - 不做开放百科大段讲解

### 实现原则

不要做“每轮都靠大模型自由发挥”。

应采用：

- **状态机**
- **固定话术模板**
- **少量参数化生成**

大模型只负责：

- 出题内容
- 简短鼓励
- 主题切换时的小变化

### 建议会话状态

为每个连接增加最小状态：

- `mode`: chat / game
- `game_type`
- `round_index`
- `correct_count`
- `topic`

### 落点文件

- `core/connection.py`
- `core/handle/textHandle.py`
- 如需拆分逻辑，新增：
  - `core/handle/game_session.py`
  - `core/handle/game_templates.py`

### Phase 2 验收标准

- 3 个游戏都能被一句话启动
- 每个游戏都能在 3 到 5 分钟内自然结束
- 游戏过程不会无限发散
- 孩子能主动重复发起其中至少 1 个游戏

---

## Phase 3：让音箱“记得你”（3 到 4 天）

### 目标

让孩子觉得它不是每次都从零开始。

### 第一步：开启最小短期记忆

当前默认 `Memory: nomem` 不适合儿童陪伴场景。

建议优先切到：

- `mem_local_short`

先只记这些内容：

- 今天玩过什么
- 最近喜欢什么主题
- 上一轮答对了几题
- 最近 1 到 3 天的高频偏好

### 第二步：开场引用记忆

系统启动时增加类似表达：

- “你昨天玩了恐龙题，今天还想玩吗？”
- “上次你猜对了 4 题，今天再挑战一下？”

### 第三步：结束时生成极短总结

结束时不做长总结，只记：

- 游戏类型
- 主题
- 正确率 / 轮数
- 孩子是否中途退出

### 落点文件

- `core/providers/memory/mem_local_short/mem_local_short.py`
- `core/utils/memory.py`
- `core/connection.py`

### Phase 3 验收标准

- 第二天使用时，系统能提到昨天玩过的内容
- 孩子能感知到“它记得我喜欢什么”
- 家长感知到互动连续性增强

## 6. 具体修改建议

## 6.1 配置层

优先通过 `data/.config.yaml` 覆盖，不直接改默认 `config.yaml`。

建议覆盖项：

- `prompt`
- `selected_module.TTS`
- `selected_module.Memory`
- `LLM.<当前模型>.max_tokens`
- 如有必要，增加儿童场景专用 `prompt_template`

## 6.2 代码层

### 必改

- `core/connection.py`
  - 增加最小会话状态
  - 注入当前模式、游戏轮次、短记忆信息
  - 控制儿童场景回复节奏

- `core/utils/prompt_manager.py`
  - 支持儿童玩伴 prompt 模板
  - 支持把主题偏好、最近游戏记录注入 prompt

- `core/providers/llm/openai/openai.py`
  - 确认儿童模式下的 `max_tokens`、流式输出行为

### 推荐新增

- `core/handle/game_session.py`
  - 管理小游戏状态机

- `core/handle/game_templates.py`
  - 放猜谜语、口算、主题问答模板

### 暂不建议动

- `core/providers/tts/huoshan_double_stream.py`
  - 该链路先单独修，不进入 V1 主路径

## 7. 验证方案

## 7.1 体验验证

找孩子连续试玩两天，每次 10 分钟内，观察：

1. 会不会主动发起
2. 中途会不会频繁说“它听不懂”
3. 会不会说它“比较笨”
4. 会不会愿意重复玩同一个主题

## 7.2 日志验证

建议额外记录以下指标：

1. ASR 结果文本
2. 首次开口耗时
3. 单轮总耗时
4. 中断发生点
5. 最常触发的游戏类型
6. 主题偏好命中情况

## 7.3 V1 是否成功的判断

满足以下 3 条即可进入 V1.5：

1. 孩子连续 3 天愿意主动使用
2. 单次互动平均超过 2 分钟
3. 家长主观评价明显提升

---

## 8. 明确延后事项

这些事情现在不要混进 V1：

1. 睡前陪伴主流程
2. Huoshan TTS 主链路切换
3. 长对话知识问答
4. 多人格、多音色复杂体系
5. 复杂长期记忆系统

## 10. 一句话结论

下一步不要把音箱继续做成“更会聊天的通用 AI”，而要把它先做成一个 **反应快、接得住、能陪 6 岁孩子玩 3 到 5 分钟的语音互动玩伴**。
